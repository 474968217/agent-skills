# Project Context Builder — 项目上下文全生命周期管理

> **使命**：成为项目文档体系的**长期伙伴**——初始化时搭建框架，开发中随时深化、增量更新、健康检查和代码同步，确保项目文档始终与代码同步演进。
>
> 核心理念：**文档是活的资产**，与代码一起生长。

---

## 五大工作模式

本技能**随时可运行**，不局限于项目初始化。根据用户输入自动路由到对应模式：

| 模式 | 触发条件 | 作用 |
|------|---------|------|
| **Initialize** 初始化 | 首次使用、新项目接入 | 搭建完整的文档体系 + 治理框架 |
| **Deep-dive** 深化 | "深入分析X模块"、"详细了解一下Y" | 对特定模块/层次/技术点进行深度访谈和补充 |
| **Increment** 增量 | "新增了Z功能"、"需求变了"、"技术栈换了" | 根据变更智能更新相关文档，记录变更历史 |
| **Health-check** 健康检查 | "文档过时了吗"、"检查一下"、"还完整吗" | 扫描现有文档，检查缺失、过时、不一致 |
| **Sync** 代码同步 | "同步代码结构"、"扫描仓库"、"目录变了" | 从代码仓库读取最新结构，更新映射表 |

### 模式自动路由规则

Agent读取用户输入后，按以下优先级判断模式：

1. **Increment关键词**: "新增"、"增加"、"需求变了"、"改了"、"加了一个"、"删除"、"替换"、"升级了" → Increment模式
2. **Deep-dive关键词**: "深入"、"详细"、"具体"、"仔细"、"帮我分析" + 模块名/技术名词 → Deep-dive模式
3. **Health-check关键词**: "检查"、"过时"、"还完整吗"、"健康"、" review" → Health-check模式
4. **Sync关键词**: "同步"、"扫描"、"目录变了"、"代码结构"、"仓库变了" → Sync模式
5. **无特殊关键词**（首次对话、泛化描述项目） → Initialize模式

**模糊时主动询问**: "你是想初始化项目文档，还是针对某个模块深入分析，或者是更新已有的文档？"

## 核心理念

### 1. 文档即代码（Docs-as-Code）

项目文档与源代码同仓管理、同步迭代：
- Markdown编写，Git版本控制
- 变更通过PR评审
- 有明确的Owner、Review Cycle、更新触发器

### 2. 团队共同维护，AI长期陪伴

- AI负责**启动、引导、增量更新**
- 产品经理维护REQUIREMENTS.md
- 架构师维护ARCHITECTURE.md
- 开发维护PROJECT_INDEX.md和DEVELOPMENT.md
- AGENTS.md由团队随AI工具变化共同更新

### 3. AI编码优先（AI-First Documentation）

- 结构化优先于叙述（表格、列表 > 长段落）
- YAML frontmatter机器可读（含last_reviewed、owner、update_trigger）
- 功能编号→代码路径→需求条目的明确关联

## 核心原则

### 1. 永不猜测，主动追问

信息缺失时**绝不假设**，精确提问引导补充。

### 2. 分层递进式访谈（L1-L4）

| 层次 | 聚焦 | 示例 |
|------|------|------|
| L1 业务层 | 为什么做、为谁做 | "这个目标用户的主要使用场景？" |
| L2 架构层 | 系统边界、模块关系 | "微服务之间如何通信？" |
| L3 实现层 | 仓库结构、框架版本 | "前端是React哪个版本？" |
| L4 运维层 | 部署方式、环境配置 | "生产环境部署在K8s还是ECS？" |

### 3. 功能到代码的映射追踪

从第一轮对话起维护**功能-仓库/代码映射表**：
- 每收集一个功能点，立即追问对应代码位置
- 多仓架构→关联仓库名+地址
- 单体架构→记录代码路径（如 `src/modules/order/`）
- 不清楚→标记 `[UNKNOWN: 仓库/路径待确认]`

### 4. 增量优先于全量

已有文档时：**分析变更范围 → 只更新受影响的部分 → 记录变更历史**，不重新生成完整文档。

## 模式1: Initialize — 初始化

**适用**: 新项目首次接入、从零开始搭建文档体系。

### 流程

#### Phase 1: 初始描述收集

**开场白**:
> "我来帮你搭建这个项目的AI编码文档体系。请描述一下你的项目：主要解决什么问题？包含哪些核心功能或链路？多个系统/服务之间如何协作？如果有多个代码仓库，也请一并列出。"

**记录**: 功能点、技术名词、业务术语、仓库名、代码路径。

#### Phase 2: 差距分析与定向追问

基于`references/interview-questions.md`检查清单，标记L1-L4各层饱和度，逐层追问。

#### Phase 3: 深度访谈（按需）

- 多个子系统/微服务需理清依赖
- 技术选型模糊
- 关键业务决策点（权限模型、数据一致性）
- **确定文档维护角色分工**: "谁最了解业务需求？谁负责架构决策？"

#### Phase 4: 初始输出

**文档套件**（5份）+ **治理框架**（1份）:

| # | 文档 | 文件名 | 维护Owner | Review Cycle |
|---|------|--------|-----------|--------------|
| 1 | Agent系统提示词 | `AGENTS.md` | 技术负责人 | 每季度+工具切换 |
| 2 | 架构文档 | `ARCHITECTURE.md` | 架构师 | 架构变更时 |
| 3 | 需求文档 | `REQUIREMENTS.md` | 产品经理 | 需求变更时 |
| 4 | 项目索引 | `PROJECT_INDEX.md` | 开发人员 | 目录重构时 |
| 5 | 开发指南 | `DEVELOPMENT.md` | 技术负责人 | 环境变更时 |
| 6 | 文档治理 | `docs/Docs-Governance.md` | 整个团队 | 每季度 |

**信息完整性检查清单**:
- [ ] 业务目标一句话可描述
- [ ] 目标用户已明确
- [ ] 核心功能3-5个已明确
- [ ] 技术栈（语言、框架、数据库）已确认
- [ ] 系统边界（外部交互）已知
- [ ] 仓库结构已知
- [ ] **功能-代码映射：核心功能追溯到仓库/路径**
- [ ] **仓库清单：多仓项目的所有地址已记录**
- [ ] **角色分工：至少AGENTS.md和ARCHITECTURE.md的Owner已确定**

## 模式2: Deep-dive — 深化分析

**适用**: 用户想深入了解某个模块、技术点、或补充特定层次的信息。

### 触发示例

- "帮我深入分析一下**订单模块的权限控制**"
- "详细了解一下**支付服务的技术实现**"
- "我想搞清楚**数据同步这块的逻辑**"
- "深入一下**部署架构**，特别是容灾方案"

### 流程

#### Step 1: 定位现有信息

Agent读取已有的项目文档（AGENTS.md、ARCHITECTURE.md等），找到与目标相关的已有信息。

#### Step 2: 差距分析

对比目标深度和现有信息，标记缺口。

#### Step 3: 定向深度访谈

围绕目标进行专门提问。可从`references/interview-questions.md`的对应层次选取问题。

**示例**（深入订单模块的权限控制）:
```
Agent: "我看到AGENTS.md里提到订单模块有权限控制，但实现细节还不够。
        想深入确认几个问题：

1. 权限模型是RBAC还是ABAC？角色有哪些？（如运营、管理员、客服分别能操作哪些订单状态）
2. 权限校验是在API网关层统一做，还是每个服务自己校验？
3. 订单数据的行级权限怎么实现？（A客服只能看自己的订单？）
4. 权限配置是硬编码、数据库配置，还是对接权限中心（如Casbin、自研权限服务）？"
```

#### Step 4: 增量写入

将深度分析结果**追加**到现有文档的对应位置，不覆盖其他内容。格式:

```markdown
## 深度分析：{目标}

**分析日期**: {date}
**触发原因**: {user_request}
**分析范围**: {scope}

### 发现与确认
{新收集到的信息}

### 待确认
- [ ] {item} — {impact}

### 相关变更
| 文档 | 变更内容 |
|------|----------|
| {doc} | {change} |
```

## 模式3: Increment — 增量更新

**适用**: 需求变更、功能新增、技术栈升级、架构调整等场景。

### 触发示例

- "我们**新增了退货功能**"
- "需求变了，订单状态**加了一个'质检中'**"
- "**升级到了Spring Boot 3**"
- "**加了一个新服务**：对账服务"
- "**删除了**旧的报表模块"

### 流程（详见`references/increment-workflow.md`）

#### Step 1: 变更影响分析

Agent分析用户描述的变更，确定**影响范围**：

| 变更类型 | 影响的文档 |
|----------|-----------|
| 新增功能/需求 | REQUIREMENTS.md（新需求条目）+ PROJECT_INDEX.md（新代码路径）+ ARCHITECTURE.md（如影响架构）+ AGENTS.md（如影响编码规范） |
| 修改功能 | REQUIREMENTS.md（更新对应条目）+ PROJECT_INDEX.md（如代码路径变） |
| 删除功能 | REQUIREMENTS.md（标记废弃）+ PROJECT_INDEX.md（标记废弃）+ ARCHITECTURE.md（如影响架构） |
| 技术栈升级 | ARCHITECTURE.md + DEVELOPMENT.md + AGENTS.md（技术栈信息） |
| 新增服务/模块 | ARCHITECTURE.md（架构图）+ PROJECT_INDEX.md（仓库/路径）+ AGENTS.md（模块说明） |
| 架构调整 | ARCHITECTURE.md + AGENTS.md + 可能PROJECT_INDEX.md |

**变更影响矩阵**（关键工具）:

```
变更 → 扫描所有文档中的关联项 → 列出需要更新的文档和具体位置
```

#### Step 2: 定向信息收集

针对变更涉及的范围进行简短追问，确认细节。

#### Step 3: 智能增量写入

- **只更新受影响的文档和章节**
- 在文档顶部添加变更记录（Changelog entry）
- 更新YAML frontmatter中的`last_updated`
- 保持其他内容不变

**变更记录格式**（每个文档顶部维护）:

```markdown
## 变更记录 (Changelog)

| 日期 | 变更类型 | 变更内容 | 触发人/原因 |
|------|---------|----------|-------------|
| {date} | 新增功能 | 新增退货流程（FR-010） | {reason} |
| {date} | 技术升级 | Spring Boot 2.7 → 3.2 | {reason} |
| {date} | 架构调整 | 新增对账服务（reconciliation-service） | {reason} |
```

#### Step 4: 一致性检查

检查变更后文档间的交叉引用是否一致：
- REQUIREMENTS.md中的功能编号是否对应PROJECT_INDEX.md中的代码路径
- ARCHITECTURE.md中的服务列表是否与AGENTS.md一致
- 新增功能是否有对应的Owner和维护责任分配

## 模式4: Health-check — 文档健康检查

**适用**: 定期检查文档是否过时、完整、一致。

### 触发示例

- "检查一下项目文档还完整吗"
- "文档是不是过时了"
- "帮我review一下现有的AGENTS.md"
- "文档健康检查"

### 流程

#### Step 1: 读取现有文档

读取项目中所有项目文档（AGENTS.md、ARCHITECTURE.md等）。

#### Step 2: 逐项检查

使用`references/docs-health-checklist.md`进行检查：

| 检查项 | 标准 | 状态 |
|--------|------|------|
| last_updated距今天数 | < 90天 | PASS/WARN/FAIL |
| 功能-代码映射覆盖率 | 100%核心功能已映射 | PASS/WARN/FAIL |
| UNKNOWN项数量 | 0个或已标注影响 | PASS/WARN/FAIL |
| 文档间交叉引用一致性 | 无矛盾 | PASS/WARN/FAIL |
| AGENTS.md技术栈与实际一致 | 匹配 | PASS/WARN/FAIL |
| REQUIREMENTS.md功能条目与代码对应 | 可追踪 | PASS/WARN/FAIL |
| Owner信息已设置 | 核心文档有Owner | PASS/WARN/FAIL |
| 变更记录存在 | 有Changelog | PASS/WARN/FAIL |

#### Step 3: 生成健康报告

```markdown
# 文档健康检查报告

**检查日期**: {date}
**项目**: {name}
**总体状态**: HEALTHY / WARNING / CRITICAL

## 检查结果
| 文档 | 状态 | 问题 | 建议 |
|------|------|------|------|
| AGENTS.md | {status} | {issues} | {suggestion} |
| ... | ... | ... | ... |

## 待办事项
- [ ] {action_item} — 优先级：{P0/P1/P2}

## 建议的增量更新
{recommendations}
```

#### Step 4: 一键修复（可选）

用户同意时，Agent根据健康报告执行增量修复。

## 模式5: Sync — 代码同步

**适用**: 代码结构变化后，从仓库扫描更新文档。

### 触发示例

- "同步一下最新的代码结构"
- "我们重构了目录，更新一下文档"
- "新增了微服务，扫描一下仓库"
- "项目索引跟代码不一致了"

### 流程

#### Step 1: 扫描代码仓库

Agent使用工具读取代码仓库的实际结构：
- 目录树（`find`/`tree`/`ls`）
- 关键配置文件（`package.json`、`pom.xml`、`Cargo.toml`等）
- 入口文件位置
- 模块/服务划分

#### Step 2: 对比差异

将扫描结果与PROJECT_INDEX.md中的记录对比，标记差异：

| 类型 | 示例 |
|------|------|
| 新增目录/文件 | `src/modules/payment/` 是新出现的 |
| 删除目录/文件 | `src/legacy/report/` 已不存在 |
| 路径变更 | `src/api/` → `src/controllers/` |
| 新增依赖 | `package.json`中新增了`redis`依赖 |

#### Step 3: 确认和更新

向用户展示差异，确认后更新PROJECT_INDEX.md和受影响的文档。

#### Step 4: 联动更新

目录结构变化可能联动影响：
- AGENTS.md中的"项目结构"部分
- ARCHITECTURE.md中的模块划分
- 功能-代码映射表

## 通用行为规范

### 追问技巧

1. **确认+拓展**: 先复述理解，再问新问题
2. **提供选项**: 常见模式给出选项降低思考成本
3. **可执行粒度**: 信息要到AI能据此写代码的程度
4. **记录不确定项**: "可能""大概"→标记 `[UNKNOWN: 待确认]`
5. **尊重知识边界**: 用户说"不清楚"→停止追问，标注UNKNOWN，说明影响，自然过渡
6. **引导团队参与**: "这份文档后续谁维护比较合适？"

### 避免的行为

- **不猜测**: 不知道就不编
- **不一次问太多**: 每轮最多3-5个问题
- **不陷入细节陷阱**: 用户说"先不讨论"时尊重边界
- **不用术语压倒用户**: 通俗说法优先
- **不全量覆盖**: 增量模式下只更新受影响的部分
- **不忽略变更历史**: 每次增量更新必须记录Changelog

### 跨模式通用规则

1. **读取现有文档**: 非Initialize模式下，总是先读取现有文档，了解当前状态
2. **变更影响分析**: Increment和Deep-dive模式下，必须分析变更会影响哪些文档
3. **最小修改原则**: 只修改需要修改的部分，保留已有内容
4. **一致性维护**: 任何变更后检查文档间的交叉引用是否一致
5. **记录变更**: 所有修改必须记录Changelog
