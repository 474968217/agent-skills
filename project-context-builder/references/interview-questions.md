# 分层提问检查清单

本文件提供按层次组织的标准化提问检查清单。Agent在访谈过程中根据信息缺口选择使用，不必全部提问，而是根据已收集信息的饱和度动态选择。

---

## L1 — 业务层（Business Context）

> 目标：理解"为什么做"和"为谁做"

### 1.1 业务目标

- [ ] 这个系统/项目主要解决什么业务问题？
- [ ] 成功的衡量标准是什么？（KPI、效率提升、成本降低等）
- [ ] 项目的优先级和紧急程度如何？
- [ ] 目前这个业务问题是用什么方式解决的？（人工/旧系统/多个系统）
- [ ] 这个项目的业务价值主要体现在哪里？

### 1.2 目标用户

- [ ] 系统的主要用户群体是谁？（C端消费者/B端企业/内部员工/开发者）
- [ ] 用户的使用场景和频率？（日常高频/偶尔使用/后台管理）
- [ ] 用户的技术水平如何？（需要简洁UI还是可以接受复杂操作）
- [ ] 是否有多个角色/权限级别？分别是什么？
- [ ] 用户主要在什么设备上使用？（PC/移动端/平板）

### 1.3 核心业务流程

- [ ] 请描述从起点到终点的完整业务链路
- [ ] 哪些步骤是高频核心流程？哪些是低频管理操作？
- [ ] 流程中需要人工审批或干预的环节有哪些？
- [ ] 异常情况的处理方式？（退款、撤销、超时、失败重试等）
- [ ] 是否有定时任务或批处理需求？

### 1.4 业务约束

- [ ] 是否有合规或监管要求？（金融、医疗、数据隐私等）
- [ ] 是否有性能或可用性承诺？（SLA、并发用户数、响应时间）
- [ ] 预算或资源方面的约束？
- [ ] 上线时间或关键里程碑？

---

## L2 — 架构层（Architecture Context）

> 目标：理解"系统长什么样"和"模块如何协作"

### 2.1 系统形态

- [ ] 整体架构是什么模式？（单体/MVC/微服务/Serverless/事件驱动）
- [ ] 系统包含哪些主要子系统/服务/模块？每个的职责是什么？
- [ ] 各模块之间的依赖关系和数据流转方向？
- [ ] 是否有分层架构？（如Controller-Service-DAO、CQRS、六边形架构）
- [ ] 系统是否涉及分布式？分布式的方式是什么？

### 2.2 技术栈

- [ ] 编程语言及版本？（Java 17/Node.js 20/Python 3.11等）
- [ ] 主要框架和运行时？（Spring Boot/Nest.js/FastAPI/Django等）
- [ ] 前端技术栈？（React/Vue/Angular/纯HTML，是否SSR）
- [ ] 数据库类型和版本？（MySQL 8/PostgreSQL 16/MongoDB 6等）
- [ ] 缓存方案？（Redis/Memcached/本地缓存，用途是什么）
- [ ] 消息队列？（Kafka/RabbitMQ/RocketMQ，使用场景）
- [ ] 搜索引擎？（Elasticsearch/Typesense/Meilisearch，用途）
- [ ] 外部依赖服务？（第三方支付/短信/邮件/OAuth等）

### 2.3 数据架构

- [ ] 核心数据模型有哪些？（用户/订单/商品等关键实体）
- [ ] 数据库设计模式？（关系型主从/分库分表/多租户隔离方案）
- [ ] 是否有大数据量场景？预估数据量级？
- [ ] 数据一致性要求？（强一致/最终一致，分布式事务方案）
- [ ] 数据迁移或同步需求？（从旧系统迁移、跨系统同步）

### 2.4 接口与集成

- [ ] 系统对外暴露的接口形式？（REST/gRPC/GraphQL/WebSocket）
- [ ] 接口认证方式？（JWT/API Key/OAuth2/AKSK）
- [ ] 与外部系统的集成方式？（Webhook/轮询/SDK/消息队列）
- [ ] 是否有开放平台或SDK需求？
- [ ] 接口文档的维护方式？

## L3 — 实现层（Implementation Context）

> 目标：理解"代码怎么组织"和"具体怎么实现"

### 3.1 代码组织

- [ ] 仓库结构是什么模式？（单体单仓/多仓/polyrepo/monorepo）
- [ ] 如果是多仓项目，所有仓库的地址/链接是什么？（GitHub/GitLab/Gitea地址）
- [ ] 每个仓库对应哪些功能/模块？（关键映射关系）
- [ ] 目录结构遵循什么约定？（按模块/按类型/按功能域）
- [ ] 代码分层或分包策略？
- [ ] 是否有共享库/公共模块？如何管理？
- [ ] 包管理工具？（Maven/Gradle/npm/pnpm/poetry）

### 3.1a 功能到仓库/代码的映射（重点）

**目标：确保每个功能/需求都能追溯到代码位置**

- [ ] 请列出项目的所有代码仓库（如果是多仓）：
  | 仓库名 | 仓库地址 | 对应功能/模块 |
  |--------|----------|--------------|
  | {repo_name} | {repo_url} | {functions} |

- [ ] 如果是单体仓库，核心功能对应的代码目录是什么？：
  | 功能/模块 | 代码路径 | 关键文件 |
  |----------|----------|----------|
  | {function} | {path} | {key_files} |

- [ ] 用户认证/鉴权相关的代码在哪个仓库/目录？
- [ ] 核心业务逻辑（如订单处理、支付流程）的代码入口在哪里？
- [ ] 前端页面的代码组织方式？（按页面/按组件/按业务域）
- [ ] 数据库模型/Entity定义在哪个位置？
- [ ] 配置文件（application.yml, .env等）在哪个位置？
- [ ] 测试代码的位置和结构？

### 3.2 核心功能实现

- [ ] 认证与授权的实现方式？（Session/JWT/OAuth2/RBAC/ABAC）
- [ ] 关键业务逻辑的核心算法或规则？
- [ ] 状态机或工作流引擎？（自研/使用框架如Activiti/Camunda）
- [ ] 文件处理需求？（上传/下载/存储位置/OSS/本地）
- [ ] 是否有实时通信需求？（WebSocket/SSE/长轮询）
- [ ] 是否有国际化/多语言需求？

### 3.3 质量保障

- [ ] 测试策略？（单元测试/集成测试/E2E测试覆盖率要求）
- [ ] 代码规范工具？（ESLint/Prettier/Checkstyle/Spotless）
- [ ] 接口测试工具？（Postman/JMeter）
- [ ] 代码审查流程？
- [ ] CI/CD流水线？（GitHub Actions/GitLab CI/Jenkins等）

### 3.4 技术债务与特殊处理

- [ ] 是否有遗留代码或历史包袱？
- [ ] 已知的技术难点或风险点？
- [ ] 是否有特殊的技术选型原因？（历史原因/团队熟悉度/生态兼容）
- [ ] 是否在迁移过程中？（新旧系统并行、灰度发布）

## L4 — 运维层（Operational Context）

> 目标：理解"系统怎么跑"和"如何维护"

### 4.1 部署与基础设施

- [ ] 部署目标环境？（公有云/私有云/混合云/本地机房）
- [ ] 容器化方案？（Docker/Kubernetes/ECS/裸机部署）
- [ ] 环境数量与用途？（开发/测试/预发/生产）
- [ ] 域名与网关配置？（Nginx/Kong/Spring Cloud Gateway）
- [ ] 静态资源托管？（CDN/OSS/服务器本地）

### 4.2 运维与监控

- [ ] 日志收集方案？（ELK/EFK/Loki/SLS/本地文件）
- [ ] 监控告警工具？（Prometheus+Grafana/CloudWatch/自建）
- [ ] 链路追踪？（SkyWalking/Jaeger/Zipkin）
- [ ] 告警通知渠道？（钉钉/企业微信/飞书/邮件/SMS）
- [ ] 备份与恢复策略？（数据库备份/配置备份/灾难恢复RTO/RPO）

### 4.3 安全

- [ ] 网络隔离策略？（VPC/安全组/防火墙规则）
- [ ] 敏感数据处理？（加密存储/传输加密/脱敏展示）
- [ ] 安全审计需求？（操作日志/登录审计/数据变更追踪）
- [ ] 漏洞管理流程？

### 4.4 团队协作

- [ ] 团队规模和角色分工？
- [ ] 开发流程？（敏捷Scrum/看板/瀑布）
- [ ] 文档管理方式？（Confluence/语雀/Notion/Markdown仓库内）
- [ ] 沟通工具？（钉钉/飞书/企业微信/Slack）

## 快速评估表

Agent在每次访谈后使用此表快速评估信息饱和度：

| 层次 | 饱和度高（>=80%） | 饱和度中（50-80%） | 饱和度低（<50%） |
|------|-------------------|-------------------|-----------------|
| L1 业务层 | 能清晰描述业务目标和核心流程 | 知道做什么但不知道为什么 | 仅有零散功能描述 |
| L2 架构层 | 技术栈、架构模式、数据流清晰 | 知道主要技术但不确定版本和交互方式 | 几乎没有架构信息 |
| L3 实现层 | 仓库结构、框架、编码规范明确 | 知道语言但不确定框架和结构 | 无实现细节 |
| L4 运维层 | 部署方式、环境配置、监控明确 | 知道大致部署方式但无细节 | 无运维信息 |

**决策规则**:
- L1必须达到"高"才能输出
- L2至少达到"中"才能输出
- L3和L4视项目阶段决定（新项目L4可低，维护项目L4应高）
