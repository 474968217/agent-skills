# 任务总结与资产沉淀

任务结束后的结构化收尾流程：信息收集 → 生成报告（含深度反问）→ 制定资产更新计划 → 用户确认后执行写入。

整个过程分为 4 个阶段，在 Phase 3 得到用户确认之前，不要修改任何文件。

## Phase 1: 信息收集与基线校验

1. 回顾当前会话中的需求、约束与补充说明。
2. 梳理本次变更范围（`git diff` 或修改文件列表），建立对实际改动的整体认知。
3. 按需读取文档基线：根目录 `CLAUDE.md`、变更涉及子目录的 `CLAUDE.md`，以及全局配置文件（如 `~/.clauderules`）。不必全量读取——只看变更可能影响的层级。
4. 将实际改动与文档基线交叉比对，重点发现两类问题：
   - 代码实现中违背现有文档规则的行为。
   - 因本次移除代码而导致失效的旧规则。

## Phase 2: 生成结构化报告

向用户呈现以下 5 个板块的报告：

### 1. 任务全景图
- **需求归纳**：背景、目标、约束条件。
- **变更总结**：从整体架构 → 核心链路 → 局部细节，逐层展开，避免流水账。
- **影响面评估**：依赖的新增/移除、环境变量变更。

### 2. 知识萃取
- 口头沟通中出现的但尚未写入代码或文档的业务逻辑。
- 踩坑记录与排雷方案——尤其是非显而易见的报错及最终解法。
- 技术决策与妥协原因。如果涉及重大架构变更，标记为 ADR 候选。

### 3. 围栏与规范演进
- **负面约束提取**：从本次任务中提炼"绝对不能做"的事项。例如："此模块绝对不能直接读数据库——必须通过 Service 层。" 负面约束的优先级始终高于正向指引——一道清晰的红线比十条建议更能防止未来的错误。
- **规则冲突检测**：明确指出代码实现与现有 `CLAUDE.md` 规则相矛盾的地方。
- **过期记忆发现**：标出因架构变更或依赖移除而不再适用的旧规则，提议清理。

### 4. 架构防腐自检
- **越界与副作用**：本次修改是否触及了预期之外的底层公共文件？是否可能对其他链路产生隐性副作用？
- **分层破坏**：是否为了快速交付绕过了 Service 层或打破了模块边界？需要明确这是临时妥协还是应固化为架构红线。
- **Agent 行为反思**：Agent 在本次任务中是否反复试错或产生幻觉？提炼值得写入记忆的"防呆指令"。

### 5. 深度反问用户

使用 `AskUserQuestion` 工具向用户弹出选项。每条反问独立编号，给出 2-4 个互斥选项。首个选项为推荐/最保守的选择。

**典型反问模板：**

| 类型 | Header | 示例问题 |
|------|--------|---------|
| 业务边界 | 业务边界 | 发现在 XX 模块的重构可能影响 YY 场景的调用方式，如何处理？ |
| 逻辑冲突 | 逻辑冲突 | 代码实现是 A 方式，但您初始描述是 B 方式，以哪个为准？ |
| 技术债/盲区 | 技术债 | 针对 XX 妥协方案 / 找不到源码的 YY 模块，如何处理？ |
| 命令授权 | 命令授权 | 本次高频使用的未授权命令（如 `go test`、`npm lint`），是否加入白名单？ |

**调用 `AskUserQuestion` 时的注意事项：**
- 每类问题有 2-4 个选项，互斥且语义自明。
- 首个选项为最保守/推荐的选择。
- 将所有反问打包为一次 `AskUserQuestion` 调用（最多 4 个问题），而不是逐条调用。
- 如果反问超过 4 条，优先保留最需要用户决策的问题，其余的降级为文本输出。

## Phase 3: 制定资产更新计划 + 最终确认

向用户提出具体的更新计划（文字输出），遵循 LCA（最近公共祖先）路由原则：

- **围栏建设**：将提取到的红线、禁区、防呆指令作为最高优先级，写入对应层级的 `CLAUDE.md` 中。
- **新增/修改**：分析知识点的作用域，定位到最近公共祖先目录。高频核心内容直接写入该层 `CLAUDE.md`；低频长篇内容写入独立的 `.md` 文件，`CLAUDE.md` 中仅保留链接索引。
- **ADR 生成**（若触发）：计划在 `docs/adr/` 下创建标准化架构决策文档，并在根 `CLAUDE.md` 建立索引。
- **修剪/删除**：列出计划从 `CLAUDE.md` 中删除的过期规则和冗余上下文。
- **记忆更新**：通用习惯 → `~/.clauderules`；项目级规范 → 根目录 `CLAUDE.md`。绝对不要把当前项目的特定业务逻辑写入全局配置文件——这会导致其他项目受到错误约束。

输出更新计划后，立刻调用 `AskUserQuestion` 进行最终确认。将 Phase 2 的反问结果和 Phase 3 的确认打包为一次调用（最多 4 个问题）。按优先级选取：

**典型确认问题：**

| 优先级 | Header | 问题 | 选项 |
|--------|--------|------|------|
| 必选 | 总结确认 | 上述总结、架构防腐自检与更新计划是否有误？ | A) 确认无误，执行更新（推荐） B) 需要修改，在 Other 中说明 |
| 必选 | 红线处理 | 如何处理本次提取的红线与禁区？ | A) 全部写入对应层级 CLAUDE.md（推荐） B) 仅写最关键的一条，其余备注 C) 本次不写入 |
| 按需 | 规则冲突 | 如何处理代码实现与文档规则的冲突？ | A) 以代码为准，更新文档（推荐） B) 以文档为准，记录待修复 C) 逐一判断 |
| 按需 | ADR 生成 | 是否需要生成 ADR 文档？ | A) 需要（推荐） B) 不需要 |

**设计原则：**
- 必选项一定有"总结确认"，其余槽位按 Phase 2 实际产出来填充——没触发 ADR 就不问 ADR，没发现规则冲突就不问冲突。
- 如果 Phase 2 的反问（业务边界/逻辑冲突/技术债/命令授权）更紧迫，优先占据槽位，确认类问题合并为一个。
- 首个选项始终是推荐/最保守的选择。

## Phase 4: 资产固化

仅在用户明确同意并回答问题后执行：

1. **结构化写入**：编辑 `CLAUDE.md` 时使用清晰的分类标签，合并去重。`CLAUDE.md` 只保留高频上下文和核心围栏规则——超过约 30 行的详细说明应剥离到独立文件，`CLAUDE.md` 只做目录索引。
2. **生成 ADR**：按"背景 → 选项 → 决策 → 后果"结构生成 ADR 文档。
3. **垃圾回收**：根据 Phase 3 确认的修剪计划，删除过期内容。
4. **复查**：对照更新计划检查——是否写入了计划中的所有内容？
5. 输出完成提示："✅ 所有文档、ADR 与围栏配置已更新并修剪完毕，本次任务正式结项！"

## 关键原则

**负面约束优先。** 在总结与写入时，"不能做什么"的优先级永远高于"应该做什么"。一条准确的红线能防止的错误，十条正向指引也未必能覆盖。

**LCA 路由。** 局部实现细节不应污染根 `CLAUDE.md`，全局架构规则不应埋在深层子目录中。把知识放在能覆盖所有受影响消费者的最紧凑作用域中。

**冲突阻断。** 存在未解决的规则冲突或未回答的深度反问时，不能进入 Phase 4。不要在文档与代码或用户意图矛盾的情况下执行写入——等用户澄清。

**`CLAUDE.md` 极简。** `CLAUDE.md` 只放高频上下文和核心围栏。某个子目录专属的内容就放在那个子目录。超过约 30 行的详细说明提取为独立的 `.md` 文件，靠链接引用。

## 示例

**输入：**
> 搞完了，刚才重构了用户认证模块，把 session 换成了 JWT，注意 jwt secret 绝对不能硬编码在代码里。

**Phase 2 输出：**

```
## 1. 任务全景图
需求: 将认证机制从 Session 迁移到 JWT
变更: auth middleware → token 签发/校验 → 移除 session 存储依赖
影响: 移除了 redis session 依赖；新增 JWT_SECRET 环境变量

## 2. 知识萃取
踩坑: jwt-go 库 v3 和 v4 的 Claims 接口不兼容，最终选用 v4 + 自定义 Claims 结构体
技术决策: 选择自定义 Claims 而非 jwt.MapClaims，因为需要强类型校验过期时间

## 3. 围栏与规范演进
红线提取: jwt secret 绝对不能硬编码，必须从环境变量读取——硬编码会导致密钥泄露且难以轮换
过期记忆: 根 CLAUDE.md 中"所有接口需携带 session cookie"规则已失效，提议删除

## 4. 架构防腐自检
越界: 本次修改意外触达了公共的 http/middleware.go，可能影响所有已注册中间件的执行顺序
分层: 无破坏
```
