### Skill Name: `task-summary`
**Description**: 任务完成后的总结与资产沉淀工具。专注于围栏建设与边界划定；自动提取负面约束；基于 LCA 原则更新文档；具备规则冲突检测、架构防腐自检及向用户的深度反问能力。

---

### 📋 Execution Instructions (执行步骤)

当用户触发此 Skill 时，严格按照以下 4 个阶段顺序执行，**在 Phase 3 结束前不得擅自修改任何文件**。

#### Phase 1: 增量信息收集与基线校验 (Gathering & Baselining)
1. 回顾当前会话上下文中的需求、约束与补充说明。
2. 梳理工作区变更（`git diff` 或修改文件列表），提取实际改动。
3. **精准读取记忆**：仅读取项目根目录、本次变更涉及子目录的 `CLAUDE.md`，以及全局配置文件（如 `~/.clauderules`）。
4. **基线校验分析**：将本次的实际代码改动与读取到的记忆基线进行交叉比对，寻找“违背现有规范的行为”和“因代码移除导致失效的旧规范”。

#### Phase 2: 生成总结报告 (Report Generation)
生成结构化报告展现给用户，包含：

*   **1. 任务全景图 (Overview & Impact)**
    *   需求归纳：背景、目标、约束。
    *   变更总结：整体架构 -> 核心链路 -> 局部细节。
    *   影响面评估：依赖增删、环境变量变更。
*   **2. 知识萃取与 ADR 触发 (Knowledge & ADR)**
    *   业务理解：代码外口述的隐性业务逻辑。
    *   踩坑与排雷：关键报错与最终解决方案。
    *   **技术决策与 ADR**：记录方案选择与妥协原因。若判定本次包含重大架构变更，需主动提议生成 ADR 文档。
*   **3. 围栏与规范演进 (Guardrails & Standards Evolution)**
    *   **红线与禁区提取（负面约束）**：提取本次任务中明确的“绝对不能做”的事项（例：“该模块绝对不能直接读取数据库”）。
    *   **规则冲突检测**：明确指出代码实现与现有 `CLAUDE.md` 规则冲突的地方。
    *   **过期记忆发现**：指出因架构变更或依赖移除而失效的旧记忆，提议清理。
*   **4. 架构防腐与自检 (Anti-Corruption & Self-Check) 🛡️**
    *   *(客观陈述本次代码修改带来的潜在架构风险)*
    *   **越界与副作用警告**：指出本次修改是否触及了预期之外的底层公共文件，或可能对其他链路产生隐性副作用。
    *   **架构防腐确认**：指出为了快速实现需求，是否破坏了现有的分层架构（如：绕过 Service 层直接操作数据），提议是否需要将其固化为架构红线。
    *   **Agent 协作反思**：反思 Agent 在本次任务中是否反复试错或产生幻觉，提炼出需要写入记忆的“防呆指令”（Agent 行为围栏）。
*   **5. 深度反问用户 (Deep QA with User) ❓**
    *   *(必须以提问的形式主动向用户发问，等待用户决策)*
    *   **业务边界发问**：发现有未覆盖或可能受影响的衍生业务场景，询问：“请问这部分场景是否需要在下期处理？”
    *   **逻辑冲突发问**：指出代码实现与用户初始描述有出入的地方，询问：“实际代码逻辑与您最初描述不一致，请问以哪个为准？”
    *   **技术债与盲区发问**：针对妥协方案、找不到的源码或无法本地验证的场景，询问：“这些盲区是否需要记录为待办事项或补充相关文档？”
    *   **命令授权发问**：列出高频使用的未授权命令，主动询问：“是否允许将这些命令加入 Agent 的执行白名单？”

#### Phase 3: 制定资产更新计划 (Asset Update Planning)
向用户提出具体的资产更新计划，**严格执行 LCA 路由与文件瘦身策略**：

*   **1. 文档更新与修剪计划（LCA 原则 & 垃圾回收）**：
    *   **围栏建设**：将提取到的“红线与禁区”、“防呆指令”作为最高优先级的规则，写入对应层级的 `CLAUDE.md` 的 `<Guardrails>` 或 `<红线>` 板块中。
    *   **新增/修改**：分析知识点影响范围，定位到最近公共祖先目录。高频核心内容写入 `CLAUDE.md`；低频长篇幅内容写入独立 `.md` 文件并仅在 `CLAUDE.md` 中留存链接索引。
    *   **ADR 生成**（若触发）：计划在 `docs/adr/` 目录下生成标准化架构决策文档，并在根目录 `CLAUDE.md` 建立索引。
    *   **修剪/删除**：列出计划从 `CLAUDE.md` 中删除的过期规则或冗余上下文。
*   **2. 记忆更新计划（作用域隔离）**：
    *   通用习惯更新至 `~/.clauderules`（或指定路径）；项目级规范写入根目录 `CLAUDE.md`。

**🛑 暂停执行：**
在此处停止，向用户输出：“请确认上述总结、**架构防腐风险点**与更新计划是否有误？**（请逐一回复【深度反问】中的问题，并指示【红线/禁区】与【规则冲突】的处理方式）**。回复后我将自动完成所有文件的更新与修剪。”

#### Phase 4: 资产固化 (Execution & Writing)
**仅在收到用户明确同意及问题解答后执行：**
1. **结构化与瘦身写入**：修改 `CLAUDE.md` 时，必须使用清晰的分类标签（特别是显眼的红线/禁区标识）进行合并去重。严禁将长篇大论直接塞入 `CLAUDE.md`。
2. **生成 ADR**：若用户同意，生成包含“背景、选项、决策、后果”的 ADR 文档。
3. **执行垃圾回收**：根据 Phase 3 确认的修剪计划，精准删除 `CLAUDE.md` 中的过期内容。
4. **文档审查**：参考neat-freak skills（如有），进行文档和审查
5. 更新全局配置文件。
6. 输出完成提示：“✅ 所有文档、ADR 与围栏配置已更新并修剪完毕，本次任务正式结项！”

---

### ⚙️ 核心约束与规则 (Constraints)
- **负面约束优先**：在总结与写入时，“不能做什么（围栏）”的优先级永远高于“应该做什么”。
- **拒绝流水账**：变更总结必须拔高到“架构与链路”层面。
- **强制 LCA 原则**：局部细节严禁污染全局 `CLAUDE.md`，全局逻辑严禁写入深层 `CLAUDE.md`。
- **`CLAUDE.md` 极简原则**：`CLAUDE.md` 仅保留“高频上下文”与“核心围栏规则”。任何超过 30 行的详细说明必须剥离到单独的 `.md` 文件中，`CLAUDE.md` 仅作目录索引。
- **冲突阻断原则**：存在未解决的“规则冲突”或未回答的“深度反问”时，必须等待用户明确表态后，才能进入 Phase 4 执行写入。
- **作用域隔离原则**：绝对禁止将当前项目的特定业务逻辑写入全局配置文件中。
