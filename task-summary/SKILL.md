# 任务总结与资产沉淀

任务结束后的结构化收尾流程：信息收集 → 生成报告（含深度反问）→ 制定资产更新计划 → 用户确认后执行写入。

整个过程分为 4 个阶段，在 Phase 3 得到用户确认之前，不要修改任何文件。

## Phase 1: 信息收集与基线校验

**触发后、进入信息收集前，先确认用户意图：**

如果用户未明确说出触发词但 agent 判断应触发（如对话中自然出现的收尾信号），使用 `AskUserQuestion` 弹窗确认（单问题，2 选项）：
- A) 是，启动结构化总结流程（推荐）
- B) 暂不总结，继续工作

如果用户明确说出了触发词，跳过此确认，直接进入信息收集。

---

1. 回顾当前会话中的需求、约束与补充说明。
2. 梳理本次变更范围（`git diff` 或修改文件列表），建立对实际改动的整体认知。如果 git diff 为空但会话中有决策产出，改为从对话中提取变更/决策列表；如果会话中也无变更（纯咨询），跳过此 step。
3. **扫描执行过程**：回顾会话中被用户阻止的操作（rejected tool calls、权限拒绝、方向纠正），以及反复出错、多次重试才成功的操作。记录每次阻止/失败的具体场景和最终解决方式——这些是未来提效的关键。
4. 按需读取文档基线：根目录 `CLAUDE.md`、变更涉及子目录的 `CLAUDE.md`，以及全局配置文件（如 `~/.clauderules`）。不必全量读取——只看变更可能影响的层级。
5. 将实际改动与文档基线交叉比对，重点发现两类问题：
   - 代码实现中违背现有文档规则的行为。
   - 因本次移除代码而导致失效的旧规则。

### 范围判断：完整模式 vs 轻量模式

在进入 Phase 2 之前，根据变更规模决定执行深度：

| 条件 | 模式 |
|------|------|
| 变更涉及 3+ 文件、或跨模块边界、或包含新功能/架构变更 | **完整模式**：Phase 2 全部 5 板块 + 反问 + Phase 3/4 全流程 |
| 纯讨论/纯文档（无代码变更）但产出了重要决策 | **轻量模式**：仅全景图 + 知识萃取（含方案记录），跳过反问和 ADR，直接询问"是否写入 CLAUDE.md？" |
| 微小改动（1-2 文件、非架构、无新功能） | **轻量模式**：仅全景图 + 知识萃取，跳过反问/ADR/围栏演进/架构自检 |

将判断结果告知用户，例如："检测到本次为微小改动（2 个文件，无架构变更），进入轻量总结模式。"

## Phase 2: 生成结构化报告

向用户呈现以下 5 个板块的报告：

### 1. 任务全景图
- **需求归纳**：背景、目标、约束条件。
- **变更总结**：从整体架构 → 核心链路 → 局部细节，逐层展开，避免流水账。
- **影响面评估**：依赖的新增/移除、环境变量变更。

### 2. 知识萃取
- 口头沟通中出现的但尚未写入代码或文档的业务逻辑。
- **功能方案记录**：如果本次变更满足以下任一条件，优先考虑记录方案设计——引入了新的对外接口、新的数据模型、新的模块依赖，或做出了非显而易见的方案选择。方案包括技术选型理由、核心链路设计、数据结构/接口约定。如果维护者看代码 5 分钟内能理解设计意图，则不需要独立方案文档。纯配置变更、UI 微调（不改交互逻辑）、单文件小改动通常不触发此条目。
- 踩坑记录与排雷方案——尤其是非显而易见的报错及最终解法。
- **执行效率沉淀**：从 Phase 1 收集的被阻止操作和反复出错的操作中提炼经验。对每一项，分析根因——是工具限制？权限配置？知识盲区？还是用户偏好不明？如果根因不明确，在 Phase 2 的深度反问中向用户确认。将确认后的经验写入防呆指令或 CLAUDE.md，目标是让下次同类工作不再踩同样的坑。
- 技术决策与妥协原因。评估是否应写入 ADR——以下情况触发：决策影响跨模块边界、涉及安全/性能/数据一致性、或产生了不可逆的后果。模块内部的工具选型（如"选了 lodash 而非手写 deepClone"）降级为 `CLAUDE.md` 中的一行备注即可，不必创建独立 ADR。

### 3. 围栏与规范演进
- **负面约束提取**：从本次任务中提炼"绝对不能做"的事项。例如："此模块绝对不能直接读数据库——必须通过 Service 层。" 负面约束的优先级始终高于正向指引——一道清晰的红线比十条建议更能防止未来的错误。
- **规则冲突检测**：明确指出代码实现与现有 `CLAUDE.md` 规则相矛盾的地方。
- **过期记忆发现**：标出因架构变更或依赖移除而不再适用的旧规则，提议清理。

### 4. 架构防腐自检
- **越界与副作用**：本次修改是否触及了预期之外的底层公共文件？是否可能对其他链路产生隐性副作用？
- **分层破坏**：是否为了快速交付绕过了 Service 层或打破了模块边界？需要明确这是临时妥协还是应固化为架构红线。
- **Agent 行为反思**：如果项目中启用了 `karpathy-guidelines` 技能，对照其四项原则（Think Before Coding / Simplicity First / Surgical Changes / Goal-Driven Execution）逐条反思本次 agent 行为。如果某个错误模式反复出现，提炼为防呆指令写入记忆。

### 5. 深度反问用户

使用 `AskUserQuestion` 工具向用户弹出选项。每条反问独立编号，给出 2-4 个互斥选项。首个选项为推荐/最保守的选择。

**典型反问模板：**

| 类型 | Header | 示例问题 |
|------|--------|---------|
| 业务边界 | 业务边界 | 发现在 XX 模块的重构可能影响 YY 场景的调用方式，如何处理？ |
| 逻辑冲突 | 逻辑冲突 | 代码实现是 A 方式，但您初始描述是 B 方式，以哪个为准？ |
| 技术债/盲区 | 技术债 | 针对 XX 妥协方案 / 找不到源码的 YY 模块，如何处理？ |
| 命令授权 | 命令授权 | 本次高频使用的未授权命令（如 `go test`、`npm lint`），是否加入白名单？ |
| 执行效率 | 执行效率 | 本次在 XX 步骤反复出错 / 被阻止，推测原因是 A，是否准确？应如何避免？ |

**调用 `AskUserQuestion` 时的注意事项：**
- 每类问题有 2-4 个选项，互斥且语义自明。
- **尽可能提供选项，而非开放式提问。** 即使问题本身是开放性的，也要给出几个最可能的选项——降低用户的决策成本。
- **总是包含一个推荐选项**（标注"推荐"），放在第一个。这对不熟悉项目细节的新手极其友好——他们可以一路回车。
- **只问那些会实质性改变计划的问题。** 跳过无关紧要的提问——每个问题都应有明确的后续行动分支。如果答案是 A 或 B 执行结果都一样，这个问题不值得问。
- 将所有反问打包为一次 `AskUserQuestion` 调用（最多 4 个问题），而不是逐条调用。
- **如果反问超过 4 条，允许分多轮进行**——而不是降级为文本输出后不管。每轮最多 4 个问题。槽位优先级（由高到低）：
  1. 逻辑冲突 / 规则冲突（涉及正确性，必须进入弹窗）
  2. 业务边界（影响后续范围）
  3. 执行效率（阻塞类的需确认根因）
  4. 技术债 / 盲区
  5. 命令授权
- 降级为文本输出的问题，在报告中用 `> **【待确认】** 问题内容` 格式标注，确保可视。在 Phase 4 入口处显式检查：降级问题是否已被用户在后续对话中回答？未回答则不能进入 Phase 4。
- 如果反问超过 4 条，优先保留最需要用户决策的问题，其余的降级为文本输出。
- **环境适配**：在 Claude Code 中使用 `AskUserQuestion` 工具；在 Codex（OpenAI）环境中使用 `request_user_input` 工具。

## Phase 3: 制定资产更新计划 + 最终确认

向用户提出具体的更新计划（文字输出），遵循 LCA（最近公共祖先）路由原则：

- **围栏建设**：将提取到的红线、禁区、防呆指令作为最高优先级，写入对应层级的 `CLAUDE.md` 中。
- **新增/修改**：分析知识点的作用域，定位到最近公共祖先目录。判断标准——如果下次 agent 在同类任务中缺少这条信息就会走弯路或犯错，则为高频核心内容，直接写入该层 `CLAUDE.md`；否则为低频，写入独立的 `.md` 文件，`CLAUDE.md` 中仅保留链接索引。功能方案文档的安置——写入变更涉及的最深层公共目录下，命名为 `DESIGN.md` 或 `{feature-name}-design.md`，在该层 `CLAUDE.md` 中以链接引用。
- **ADR 维护**：主动检查 `docs/adr/` 现有文档。列出需要创建、更新或废弃的 ADR：
  - **新建**：本次做出的值得记录的技术决策（方案选型、依赖引入、架构约束）
  - **更新**：本次变更修改了某个已有 ADR 中记录的决策——该 ADR 需要追加修订段落
  - **废弃**：本次变更使某个 ADR 不再适用——标记 `[SUPERSEDED]` 并指向替代文档，**保留原文件而非删除**（历史决策仍有参考价值）
- **修剪/删除**：列出计划从 `CLAUDE.md` 中删除的过期规则和冗余上下文。
- **记忆更新**：通用习惯 → `~/.clauderules`；项目级规范 → 根目录 `CLAUDE.md`。绝对不要把当前项目的特定业务逻辑写入全局配置文件——这会导致其他项目受到错误约束。

输出更新计划后，**如果当前项目中存在 `neat-freak` 技能，先调用它执行洁癖级审查**——检查本次计划的文档更新是否与代码一致、写入位置是否合理、是否有遗漏或冲突。将 neat-freak 的修正建议直接纳入更新计划中（标注 `[neat-freak]` 来源），再统一呈现给用户确认。这样用户一次性审阅完整计划，不存在事后覆盖。

然后调用 `AskUserQuestion` 进行最终确认。将 Phase 2 的反问结果和 Phase 3 的确认打包为一次调用（最多 4 个问题）。按优先级选取：

**典型确认问题：**

| 优先级 | Header | 问题 | 选项 |
|--------|--------|------|------|
| 必选 | 总结确认 | 上述总结、架构防腐自检与更新计划是否有误？ | A) 确认无误，执行更新（推荐） B) 需要修改，在 Other 中说明 |
| 必选 | 红线处理 | 如何处理本次提取的红线与禁区？ | A) 全部写入对应层级 CLAUDE.md（推荐） B) 仅写最关键的一条，其余备注 C) 本次不写入 |
| 按需 | 规则冲突 | 如何处理代码实现与文档规则的冲突？ | A) 以代码为准，更新文档（推荐） B) 以文档为准，记录待修复 C) 逐一判断 |
| 按需 | ADR 生成 | 是否需要生成 ADR 文档？ | A) 需要（推荐） B) 不需要 |

**设计原则：**
- 必选项一定有"总结确认"，其余槽位按 Phase 2 实际产出来填充——没触发 ADR 就不问 ADR，没发现规则冲突就不问冲突。
- 如果 Phase 2 的反问（业务边界/逻辑冲突/技术债/命令授权）更紧迫，优先占据槽位，确认类问题合并为一个。
- 首个选项始终是推荐/最保守的选择。

## Phase 4: 资产固化

仅在用户明确同意并回答问题后执行：

1. **结构化写入**：编辑 `CLAUDE.md` 时使用清晰的分类标签，合并去重。`CLAUDE.md` 只保留高频上下文和核心围栏规则——超过约 30 行的详细说明应剥离到独立文件，`CLAUDE.md` 只做目录索引。
2. **维护 ADR**：按 Phase 3 计划执行。如果 `docs/adr/` 目录不存在，先创建该目录。同时检查根 `CLAUDE.md` 是否有 ADR 索引段落——如果没有，本次一并创建。新建 ADR 时先检查已有 ADR 的命名格式并沿用。项目尚无 ADR 则默认使用 `ADR-{序号:04d}-{短标题}.md`（4 位零填充，如 `ADR-0003-auth-jwt.md`）。按"背景 → 选项 → 决策 → 后果"结构生成。更新根 `CLAUDE.md` 的 ADR 索引。废弃旧 ADR 时在文档顶部标注 `[SUPERSEDED by ADR-XXXX]`，保留原文件。
3. **垃圾回收**：根据 Phase 3 确认的修剪计划，删除过期内容。
4. **复查**：对照更新计划检查——是否写入了计划中的所有内容？
5. 输出完成提示："✅ 所有文档、ADR 与围栏配置已更新并修剪完毕，本次任务正式结项！"

## 关键原则

**负面约束优先。** 在总结与写入时，"不能做什么"的优先级永远高于"应该做什么"。一条准确的红线能防止的错误，十条正向指引也未必能覆盖。

**方案可追溯。** 每个新功能的技术方案都应有书面记录。代码告诉你"怎么做"，方案告诉你"为什么这样做"——缺少后者，后续维护者只能从代码反向猜测设计意图，容易误改。方案放在功能模块最近的 `CLAUDE.md` 或独立的设计文档中。

**ADR 即历史。** ADR 不是一次性的归档文件，而是持续演进的决策日志。每次任务结束时主动检查：本次变更是否产生了新决策？是否修改了旧决策？是否让某个旧决策过时？废弃的 ADR 保留原文、标注指向——删除历史会让后来者无法理解当初为什么走了某条路。

**LCA 路由。** 局部实现细节不应污染根 `CLAUDE.md`，全局架构规则不应埋在深层子目录中。此原则适用于所有文档资产（`CLAUDE.md`、方案文档、ADR 索引等），不只是 `CLAUDE.md`。路由步骤：
1. 计算本次变更涉及的所有目录的树形 LCA（最近公共祖先）。
2. 检查 LCA 层级是否已有 `CLAUDE.md` —— 若有，写入该文件。
3. 若无，向上查找最近的有 `CLAUDE.md` 的层级。判断知识是否属于该层级的全局作用域：
   - 是全局性约束（如"所有模块禁止直连数据库"）→ 写入找到的 `CLAUDE.md`。
   - 不是全局性的（仅影响 LCA 子树）→ 在 LCA 层级新建 `CLAUDE.md`。
4. 根目录 `CLAUDE.md` 永远是有效的兜底层级。

**冲突阻断。** 存在未解决的规则冲突或未回答的深度反问时，不能进入 Phase 4。不要在文档与代码或用户意图矛盾的情况下执行写入——等用户澄清。

**`CLAUDE.md` 极简。** 两层控制：(1) 单个知识条目超过约 30 行时提取为独立 `.md` 文件，`CLAUDE.md` 中仅保留链接索引；(2) `CLAUDE.md` 文件总长应 ≤ neat-freak 推荐的 ~300 行 / ~15KB 软限制（若项目中无 neat-freak，按此标准自行约束）。某个子目录专属的内容就放在那个子目录。

## 示例

**输入：**
> 搞完了，刚才重构了用户认证模块，把 session 换成了 JWT，注意 jwt secret 绝对不能硬编码在代码里。

**Phase 2 输出：**

## 1. 任务全景图
需求: 将认证机制从 Session 迁移到 JWT
变更: auth middleware → token 签发/校验 → 移除 session 存储依赖
影响: 移除了 redis session 依赖；新增 JWT_SECRET 环境变量

## 2. 知识萃取
踩坑: jwt-go 库 v3 和 v4 的 Claims 接口不兼容，最终选用 v4 + 自定义 Claims 结构体
方案记录: JWT 认证方案——token 签发链路（login → generateToken → sign with HS256 → 返回 access+refresh token），校验链路（middleware → extract Bearer → verify signature → check expiry → inject claims to context）。选择 HS256 而非 RS256 是因为当前为单服务部署，无需分布式公钥分发；若未来拆分为微服务，需迁移到 RS256（见 ADR-0003）。
技术决策: 选择自定义 Claims 而非 jwt.MapClaims，因为需要强类型校验过期时间
ADR 评估: 本次引入了新的外部依赖（golang-jwt/v5）且修改了认证中间件的核心链路——触发 ADR-0003

## 3. 围栏与规范演进
红线提取: jwt secret 绝对不能硬编码，必须从环境变量读取——硬编码会导致密钥泄露且难以轮换
过期记忆: 根 CLAUDE.md 中"所有接口需携带 session cookie"规则已失效，提议删除

## 4. 架构防腐自检
越界: 本次修改意外触达了公共的 http/middleware.go，可能影响所有已注册中间件的执行顺序
分层: 无破坏

## 5. 深度反问 → 调用 AskUserQuestion
